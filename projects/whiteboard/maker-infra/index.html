Standardize

What's unique

The storyCopy, headlines, emotional hook. "365 days of your baby" vs "Grandma, Est. 2026" — totally different vibes.
Visual designColor palette, photography, layout. Each maker gets its own look.
Target audienceNew parents, gift buyers, art lovers — determines everything about the page.

What could be shared

Page shellMeta tags, analytics, font loading, responsive breakpoints. maker-shell.js
CTA patterns"Make yours" button styles, scroll-to-upload behavior, sticky mobile CTA bar.
Social proofReview widget, "X made this week" counter, trust badges.

Open questions

Do makers live on SPS or standalone?socialprintstudio.com/make/X vs. standalone URLs? Affects branding.
Ben's design system?Does Ben want a shared Figma component library for hook pages, or total creative freedom per maker?

Build once

Drag & drop zoneDesktop file drop with visual feedback. <sps-upload>
Mobile cameraTap to open camera or photo library. Native-feeling on iOS/Android.
Multi-selectPick 12 photos for a calendar, 38 for a magazine, 6 for a cube. Configurable count.
Client-side resizeResize to max dimension before upload. Saves bandwidth, faster UX.

Config per maker

Photo countmin: 6, max: 365, recommended: 12 — varies per product.
Aspect ratio hintsSquare crop for magnets, portrait for calendars, landscape for prints.
Upload messaging"Add your 12 best photos" vs "Upload a photo of the happy couple"

Implementation

Single JS componentOne sps-uploader.js that any maker imports. Web component or vanilla JS module.
Events-basedEmits events: upload:start, upload:complete, upload:error. Maker listens and reacts.
CDN uploadStraight to S3/Cloudflare. Returns URL array. No server round-trip needed for preview.

Standard components

Size pickerGrid of product sizes with prices. Pulls from PrintKit API. <sps-sizes>
Material selectorWood / metal / acrylic / paper toggle. Standard UI for format products.
Quantity stepper+/- with bulk pricing display if applicable.

Product-specific

Calendar: date rangeStart month picker, birth date input. Unique to calendar products.
Photo arrangementDrag to reorder, auto-layout, manual grid placement. Varies wildly per product.
Text personalization"NANA — Est. 2026" or baby name input. Different fields per maker.

The unique interaction

This is the maker's soulThe FaceCube grid alignment. The growth strip photo placement. The book page editor. This can't be templated.
Rule of thumbIf it's about SELECTING options → standardize it. If it's about CREATING the layout → it's custom.

Shared renderer

Canvas compositingTake user photos + product template → render preview image. sps-preview.js
3D product mockupsFlat print on wall, frame with mat, book on table. Reusable scene templates.
Zoom & rotatePinch to zoom, click to see detail. Standard interaction layer on top of any preview.

Default templates

Wall sceneProduct hanging on a styled wall. Works for prints, frames, metal, wood, canvas.
Desk/shelf sceneProduct on a styled surface. Works for books, blocks, calendars, cubes.
Flat layTop-down product shot. Works for magnets, stickers, calendars, strips.

Custom previews

Product-specific magicCalendar flipping animation. Cube rotation. Book page turn. Growth strip on a fridge door. These sell the product.
The preview IS the conversionThis is where someone decides to buy. Invest in making it feel real and delightful.

Fully standard

PrintKit order APIPOST /orders — product, variants, photos, shipping. One endpoint for everything.
Shopify cart bridgeAdd to existing Shopify cart with line item properties (photo URLs, customization data).
Price displayPull pricing from PrintKit catalog. Show base + options. sps-price.js

Checkout flow

Option A: Shopify redirectAdd to cart → redirect to Shopify checkout. Simplest. Customer sees normal SPS checkout.
Option B: Embedded checkoutStripe/Shopify checkout embedded in maker page. Seamless but more work.
Option C: PrintKit directPrintKit handles payment. Standalone from SPS Shopify. For third-party makers.

Decision needed

Which checkout path?Option A is fastest to build. Option C is needed for PrintKit-as-platform. Can start with A and add C later?
Cart vs. direct buyDo makers add to a cart (cross-sell opportunity) or go straight to checkout (higher conversion)?

Open Questions — Discuss on the TV
Question 01
Web components or plain JS modules?
<sps-upload> custom elements vs. new SPSUploader(el, config). Web components are more portable but heavier. Plain JS is simpler. What's Alejo's preference?
Question 02
Do makers live on socialprintstudio.com or standalone?
SPS domain gets SEO + trust. Standalone URLs are cleaner for ads + sharing. Could do both: SPS-hosted for internal makers, standalone for PrintKit third-party makers.
Question 03
How opinionated is the brand kit?
Does every SPS maker use Instrument Serif + Hanken Grotesk + the terracotta palette? Or is each maker free to have its own visual identity? Ben — what's the brand flexibility here?
Question 04
Cart add vs. direct checkout — what converts better?
Cart = cross-sell opportunity ("add magnets too?"). Direct = fewer steps to purchase. We could A/B this later, but need a default for v1. What's SPS's current checkout flow tell us?
Question 05
Build sps-preview now or later?
The preview renderer is the hardest shared component. Canvas compositing, 3D mockups, animations. Is this a build-week thing, or do we hard-code previews in each maker for now and extract later?
Question 06
What's the discovery UX when we have 10+ makers?
Intent-based entry ("I need a gift") vs. product gallery vs. quiz flow. Not a build-week problem, but shapes how we think about the maker ecosystem. Park it or sketch it?
